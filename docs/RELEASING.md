# Releasing

This guide lists the release stages for the `@shlomoa/mat-image-carousel` package, from local validation through GitHub tagging/release creation and publishing to npmjs.org.

## Release artifacts

- Demo app build output: `dist/photo-carousel-demo/`
- Library build output: `dist/mat-image-carousel/`
- npm package tarball: `dist/mat-image-carousel/shlomoa-mat-image-carousel-<version>.tgz`
- GitHub tag/release: generated by `.github/workflows/release.yml`
- npm publish workflow: `.github/workflows/npm-publish.yml`
- npm package: `@shlomoa/mat-image-carousel`

## 1. Prepare the branch

1. Start from `main` and pull the latest changes.
2. Confirm the working tree is clean.
3. Review the library version in `projects/mat-image-carousel/package.json`.
   - This version is the npm package version.
   - Update it before publishing if the next npm release should use a new version.
4. Review release notes or notable changes that should appear in the GitHub Release generated notes.

## 2. Install and validate dependencies

Install from the lockfile before release validation:

```bash
npm ci
```

This ensures the release checks use the same dependency graph as CI.

## 3. Run quality gates

Run the full local validation set from the workspace root:

```bash
npm run lint
npm test
npm run build
npm run build:lib
```

Expected outputs:

- `npm run lint` passes for both `photo-carousel-demo` and `mat-image-carousel`.
- `npm test` passes for both the demo app and library test suites in Chrome Headless.
- `npm run build` writes the production demo bundle to `dist/photo-carousel-demo/`.
- `npm run build:lib` writes the library package output to `dist/mat-image-carousel/`.

## 4. Pack and inspect the library package

Create a local npm tarball:

```bash
npm run pack:lib
```

This runs the library build and then `npm pack` inside `dist/mat-image-carousel/`.

Before publishing, inspect the generated tarball contents:

```bash
cd dist/mat-image-carousel
npm pack --dry-run
```

Confirm the package includes only the expected library files, metadata, typings, and bundles.

## 5. Create the GitHub tag and release

The repository includes a manual GitHub Actions workflow at `.github/workflows/release.yml`.

To create the tag and GitHub Release:

1. Open the repository on GitHub.
2. Go to **Actions**.
3. Select **Release and tag**.
4. Choose **Run workflow** on `main`.

The workflow stages are:

1. Checkout the repository with full history and tags.
2. Mark `scripts/bump_tag_main_latest.sh` executable with `chmod +x`.
3. Run `scripts/bump_tag_main_latest.sh`.
   - The script finds the latest tag merged into `HEAD`.
   - It increments the final numeric suffix.
   - It creates an annotated tag.
   - It pushes the new tag to `origin`.
4. Export the computed tag as `GITTAGVER`.
5. Create a GitHub Release for that tag with `gh release create --generate-notes`.

After the workflow completes, verify:

- The new tag exists in GitHub.
- The GitHub Release exists and has generated notes.
- The tag points at the intended commit.

## 6. Publish to npmjs.org

The preferred npm publishing path is the manual GitHub Actions workflow at `.github/workflows/npm-publish.yml`.

To publish to npmjs.org:

1. Confirm the repository secret `NPM_TOKEN` is configured with an npm automation/access token that can publish `@shlomoa/mat-image-carousel`.
2. Open the repository on GitHub.
3. Go to **Actions**.
4. Select **Node.js Package**.
5. Choose **Run workflow** on `main`.
6. Set the `npm-tag` input.
   - Use `latest` for the normal stable release.
   - Use another dist-tag, such as `next` or `beta`, for prerelease validation.

The workflow stages are:

1. Checkout the repository with full history and tags.
2. Set up Node.js 24.
3. Run `npm ci`.
4. Run `npm run build:lib`.
5. Run `npm run lint`.
6. In the publish job, set up Node.js 24 with `registry-url: https://registry.npmjs.org/`.
7. Run `npm ci`.
8. Run `npm run pack:lib`.
9. Publish from `dist/mat-image-carousel` with `npm publish --tag "<npm-tag>"` using `NODE_AUTH_TOKEN: ${{ secrets.NPM_TOKEN }}`.

Requirements:

- The `NPM_TOKEN` GitHub secret must be valid and authorized to publish the package.
- The `version` in `projects/mat-image-carousel/package.json` must not already exist on npm.
- The package uses `publishConfig.access: public`, so it publishes as a public scoped package.

After the workflow completes, verify the package and dist-tag on npmjs.org.

### Local publish fallback

If you need to publish manually from a local machine, use:

```bash
npm run publish:lib
```

This command:

1. Builds the library with `npm run build:lib`.
2. Changes directory to `dist/mat-image-carousel`.
3. Runs `npm publish`.

This requires an authenticated local npm session for an account that can publish `@shlomoa/mat-image-carousel`.

To verify npm authentication before publishing:

```bash
npm whoami
```

If publishing fails because the version already exists, update `projects/mat-image-carousel/package.json`, commit the version bump, rerun validation, and publish again.

## 7. Post-release verification

After publishing:

1. Confirm the package appears on npmjs.org:

```bash
npm view @shlomoa/mat-image-carousel version
```

2. Confirm the published version matches `projects/mat-image-carousel/package.json`.
3. Optionally test installation in a separate Angular workspace:

```bash
npm install @shlomoa/mat-image-carousel
```

4. Confirm the GitHub Release and npm package version are documented in any project notes or changelog process used for the release.

## Recovery notes

- If the GitHub Release workflow fails before creating a tag, fix the issue and rerun it.
- If the workflow creates/pushes a tag but fails to create the GitHub Release, create the release manually from that tag or rerun only after confirming the script will not create an unintended extra tag.
- If the `Node.js Package` workflow fails before publishing, fix the issue and rerun it with the same `npm-tag`.
- If npm publishing fails after the GitHub Release is created, fix the npm issue and rerun the `Node.js Package` workflow; do not recreate the GitHub tag unless the released source commit changes.
- If local `npm publish` is used as a fallback, verify that the same package version and dist-tag are published as intended.
